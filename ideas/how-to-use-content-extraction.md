# How to Use Content Extraction — 操作指南

> 创建日期：2026-06-06 | 持续更新中
>
> **定位**：本文是 [content-extraction.md](./content-extraction.md) 的**使用指南**，专注于「拿到 raw/ 后怎么做」。
> content-extraction.md 是工具手册（怎么获取内容），本文是使用手册（怎么分析内容、构建知识）。
>
> **关联文档**：
> - [content-extraction.md](./content-extraction.md) — 各来源提取工具和脚本
> - [content-extraction-tool-plan.md](./content-extraction-tool-plan.md) — 工程化落地设计

---

## 整体流程图

```
raw/*.md                →           wiki/*.md              →       你的查询
（TUI/CLI 生成的原料）       （Claude Code 整理的结构化笔记）     （自然语言提问）

       ↑                          ↑              ↑
  content-extract            CLAUDE.md        /content-extract
  （本文不涉及）          （极简项目结构声明）   （Skill：所有分析逻辑）
```

**核心分工**：
- `raw/` 是机器生产的，格式一致，人不读
- `wiki/` 是 Claude 整理的，人通过 Obsidian 消费
- `CLAUDE.md` 只声明项目结构（极简，约 15 行），不含分析策略
- `Skill` 包含所有判断逻辑：检测内容类型、选对应策略、困惑驱动等
- 你只做两件事：**策源**（决定抓什么）和**提问**（决定问什么）

# Part 1：基础流程

## 1.1 建立 CLAUDE.md（一次性）

CLAUDE.md 是你和 Claude 之间的「合同」——告诉它你是谁、内容是什么、wiki 该怎么组织。

放在项目目录的根（`raw/` 的上级），Claude Code 启动时自动读取。

### 通用模板

```markdown
# 知识库构建规则

你是这批内容的知识管理员。
原始内容在 `./raw/` 目录（Markdown 格式）。
任务：构建结构化 Wiki 到 `./wiki/`。

## 内容来源识别

raw/ 文件名前缀说明：
- `web__`：网页/博客爬取
- `bili__`：Bilibili 视频转录
- `article__`：单篇网络文章

每个文件的 frontmatter 包含 source / type / platform 等字段。

## Wiki 目录结构

wiki/
├── INDEX.md          # 所有概念索引 + 来源分布图
├── concepts/         # 核心概念（跨来源整合）
│   └── CONCEPT.md
├── by-source/        # 按来源分类的摘要
└── changelog.md

## 每个概念页面格式（wiki/concepts/CONCEPT.md）

---
sources: [文件名列表]
related: [[concept-a]], [[concept-b]]
last-updated: YYYY-MM-DD
---

## 核心定义
（1-3 句话）

## 关键细节
（技术细节、具体数据、重要案例）

## 不同来源的视角
（同一概念在不同视频/文章中的表述差异）

## 未解答的问题
（整理这个概念时仍不清楚的地方）

## 构建步骤

1. 读取所有 `./raw/*.md` 文件
2. 识别跨来源的核心概念，每个建一个 `concepts/CONCEPT.md`
3. 为每种来源类型生成摘要（`by-source/*.md`）
4. 构建 `INDEX.md`：概念列表 + 来源分布 + 关联关系

## 更新规则

新文件加入 raw/ 时：
- 更新受影响的概念页面
- 在 changelog.md 记录
- 如有新概念，添加到 INDEX.md
```

> **建议**：不同内容类型要定制 CLAUDE.md。投资视频和技术文档的整理逻辑完全不同。
> 见 Part 3 各内容类型的具体 CLAUDE.md 配置。

## 1.2 第一次构建 Wiki

```bash
# 进入项目目录
cd /path/to/your/project

# 启动 Claude Code
claude
```

在 Claude Code 里，**先做快速扫描，确认方向再全量建**：

```
第一步（扫描）：
读取 ./raw/ 下所有文件的文件名和 frontmatter，
告诉我：有多少个文件、主要内容类型是什么、你识别出的核心主题有哪些。
不用读正文，只看元数据。
```

确认方向对了，再：

```
第二步（全量建）：
读取 ./raw/ 下所有文件，按照 CLAUDE.md 的规则构建 Wiki 到 ./wiki/。
先建 INDEX.md 框架，再逐个建 concepts/ 页面。
```

## 1.3 日常查询

Wiki 建好后，三种查询方式按需选择：

### 方式 A：直接问 Claude Code（最常用）

```bash
claude
```

```
根据 wiki/，[主题] 的核心逻辑是什么？
```

```
wiki/concepts/ 里哪些概念是相互矛盾的？找出来并解释。
```

```
我想了解 [概念X]，从 wiki 里找出所有相关内容，
给我一个从基础到深入的阅读路径。
```

### 方式 B：Obsidian 浏览（通勤/碎片时间）

把 `wiki/` 目录用 Obsidian 打开：
- Graph view 看概念关联网络
- DASHBOARD.md 看最近更新和知识 gap
- 双击 `[[链接]]` 在概念间跳转

### 方式 C：带上下文的精准查询

```bash
# 先找相关文件
grep -rl "关键词" ./wiki/

# 在 Claude Code 里：
# "读取这几个文件，深入解释 XXX"
```

## 1.4 新内容加入时更新

每次 TUI/CLI 抓完新内容后，在 Claude Code 里：

```
raw/ 有新增文件：[文件名列表]
请更新 wiki/：
- 补充/更新相关 concepts/ 页面
- 在 changelog.md 记录本次新增
- 如有新概念，添加到 INDEX.md
不需要重建整个 wiki，只更新受影响的部分。
```

## 1.5 常见问题处理

### 问题：raw/ 文件太多，一次塞不进上下文

Claude 的上下文约 200k token ≈ 600KB 文本。文件过多时分批处理：

```
先只读 raw/ 下前 20 个文件，建立初始 wiki 框架和主要概念列表。
```

```
继续读接下来 20 个文件，补充和更新 wiki。
（已有的 concepts/ 文件你可以直接修改，不需要重建）
```

### 问题：不知道 wiki 里有什么

```
读取 wiki/INDEX.md，告诉我当前 wiki 里有哪些主要概念，
按「出现频次」和「关联数」排序，并标出哪些概念是孤立的（没有 related）。
```

### 问题：wiki 质量参差不齐（概念太碎）

```
读取所有 wiki/concepts/ 文件，
找出哪些概念实际上是同一个东西的不同表述，建议合并列表。
哪些概念只出现在 1 个来源里，标记为「低置信度」。
```

# Part 2：进阶技巧

## 2.1 困惑驱动的第二轮提取

> 适用于：第一轮构建完 wiki 后，发现还有大量「不知道自己不知道」的盲区。

**原理**：wiki 建好后，让 Claude 主动暴露它的盲区，再用这些问题驱动定向提取。比「尽量多抓内容」更精准。

```
第一步：
读取当前所有 wiki/concepts/ 文件。
作为一个需要深度理解这批内容的人，
列出你还不清楚的 10 个关键问题。
格式：问题 → 可能在哪里找到答案（什么类型的内容/来源）
```

Claude 会输出类似：

```
1. 作者对「抄底」和「趋势跟随」的优先级判断是什么？→ 可能在具体策略类视频
2. 这些建议适用于什么仓位规模？有没有提到资金门槛？→ 可能在实战案例类视频
3. 风险管理的具体操作（止损位置、仓位比例）在哪里有详细说明？→ ...
```

```
第二步：
根据问题1，在 raw/ 里搜索相关内容：
grep -rl "抄底\|趋势" ./raw/
把找到的文件读一遍，补充 wiki/concepts/ 里的相关页面。
```

重复直到 Claude 不再产生新问题。

---

## 2.2 跨来源对比分析

当你有多个来源（多个 UP 主、视频 + 文章等）时：

```
读取 wiki/by-source/ 下所有文件，
比较不同来源对「[同一主题]」的观点差异：
- 有哪些一致的观点（可信度高）
- 有哪些明显矛盾的地方（需要我自己判断）
- 哪个来源对这个主题的解释最系统完整
```

---

## 2.3 发现知识 gap

```
读取 wiki/INDEX.md 和 wiki/DASHBOARD.md（如果存在）。
告诉我：
1. 哪些概念有「未解答的问题」（frontmatter 里的该字段不为空）
2. 哪些概念是孤立的（没有 related 连接）
3. 按重要性排序，列出最值得补充的 5 个 gap
```

# Part 3：不同内容类型的处理差异

> 这一 Part 会持续扩充。每种内容类型的特点不同，CLAUDE.md 的配置和查询策略也不同。

---

## 3.1 投资/财经视频（Bilibili）

**内容特点**：
- 口语化转录，有重复和填充词
- 同一概念被多个视频反复提及（信号强度不同）
- 有时效性（市场观点随时间变化）
- 有些视频是「故事化」的，不是「方法论」的

**CLAUDE.md 定制要点**：

```markdown
## 内容特征
- 全部是投资/炒股主题的短视频口语转录
- 字幕有重复行和填充词（已预处理简体中文，但口语特征保留）
- 核心价值：反复出现的观点 > 只提一次的观点
- 注意时效性：涉及具体市场判断的内容标注日期

## 概念分类（wiki/concepts/ 下按这三类组织子目录）
- 理念/哲学类：对投资本质的看法（不受时效影响）
- 方法论类：具体操作策略（需注明适用条件）
- 案例类：具体市场分析（强时效性，标注日期）

## 重要提示
- 「谷票」「谷市」是「股票」「股市」的口语谐音写法，统一处理为「股票/股市」
- 重复出现 3 次以上的观点才建立独立概念页面
- 相互矛盾的观点要在 INDEX.md 底部的「冲突记录」里列出
```

**推荐的第一个查询**：

```
读取所有 bili__*.md 文件，
统计每个核心概念（投资理念/操作策略）在多少个视频中出现过，
按出现频次排序列出 Top 15，这是最值得建 wiki 页面的概念。
```

**特别有用的查询**：

```
这位 UP 主对「散户」的核心判断是什么？
他认为散户最常犯的 3 个错误是什么，在哪些视频里有具体说明？
```

```
把他关于「什么时候该空仓」的所有表述汇总，
看看他的标准是否一致，还是在不同视频里有不同说法。
```

## 3.3 代码工程

**内容特点**：
- 目标是「理解架构和修改意图」，不是「记录所有代码」
- 大型工程（万行以上）全量提取耗时长且 LLM 上下文放不下，需要分层策略
- 最有价值的信息不是代码行本身，而是**模块间关系**和**历史变更模式**
- 不同规模工程适合不同的提取深度（见 3.7 节通用落地指南）

**三档提取模式**（content-extract TUI 选 `code 本地代码工程`）：

| 模式 | 输出 | 耗时（以 spartacus 为例） | 适用场景 |
|------|------|--------------------------|---------|
| `overview`（默认） | 1 个 `__overview.md` | ~0.3s | 快速建立全局认知 |
| `priority` | ~92 个文件 + `__imports.json` | ~1s | 准备修改代码前 |
| `full` | ~12098 个文件 + `__imports.json` | ~68s | 深度阅读特定模块 |

**`__overview.md` 包含的内容**（overview 模式自动生成）：
- **架构层识别**：启发式匹配目录名，识别 API层 / 业务逻辑层 / 数据访问层 / UI层 / 工具层
- **推荐阅读路径**：基于配置文件、入口文件、git 热力图自动生成 5 步阅读顺序
- 目录结构（3 层深）、配置文件全文、README、git log（最近 30 条）、高频变更文件 Top 20

**`__imports.json` 包含的内容**（priority / full 模式生成）：
- 已提取文件的 import 依赖关系图，格式：`{ "src/auth/login.ts": ["../auth.service", "../../shared/models"] }`
- 让 LLM 能直接看出模块调用链，而不是逐文件猜测

**CLAUDE.md 定制要点**：

```markdown
## 内容特征
- 这是一个 [语言/框架] 代码工程的提取结果
- `__overview.md`：项目全局概览（架构层、阅读路径、git 信息）
- `__imports.json`：模块 import 依赖图（priority/full 模式生成）
- 测试文件（`*.spec.ts` / `test_*.py`）优先阅读——它们是系统契约文档
- 目标：理解架构和核心模块职责，不是记录每行代码

## 分析目标
- 理解系统整体架构（模块边界、数据流向、关键依赖）
- 识别高频变更模块（复杂度热点）
- 找到新开发者需要最先了解的 3-5 个核心文件
```

**推荐的第一个查询**（overview 模式后）：

```
读取 ./raw/<工程名>/__overview.md，
基于其中的架构层识别和推荐阅读路径，
用三个层次回答：
1. 一句话：这个工程是什么
2. 五分钟解释：主要模块、数据流向、核心文件
3. 深度拆解：各模块职责、模块间关系、新人应该先看什么
只陈述文件中有的事实，不推断和建议重构。
```

**priority 模式后的查询**（读测试 + 类型 + imports）：

```
读取 ./raw/<工程名>/__imports.json，
找出被 import 次数最多的 5 个模块，
这些是系统的核心依赖枢纽。
再读这 5 个模块对应的 .md 文件，解释它们各自负责什么。
```

```
读取所有 *.spec.ts / *.test.ts 文件，
列出这个系统对外承诺的核心契约（what it should do）：
- 有哪些主要功能被测试覆盖？
- 有哪些边界条件被明确处理？
- 有哪些明显没有测试覆盖的区域？
```

> **大型工程（>5000 个文件）的处理策略**：见 3.7 节通用落地指南。

## 3.7 通用代码工程落地指南

> 以下流程适用于任何代码工程（Angular/TypeScript、Python、Go、Java 等），
> 将「抓取 → 理解 → 深入」分为四个阶段，按工程规模选择进入深度。

---

### 阶段 0：判断工程规模，选择策略

先在 TUI 里用 `overview` 模式跑一次（~0.3s），读 `__overview.md` 中的目录结构和文件数量，
然后按下表选择后续策略：

| 工程规模 | 文件数量参考 | 推荐策略 |
|---------|------------|---------|
| **小型**（脚本/库） | < 500 个文件 | overview → 直接 full → wiki 构建 |
| **中型**（服务/应用） | 500–5000 个文件 | overview → priority → 困惑驱动定向补充 |
| **大型**（平台/框架） | > 5000 个文件 | overview → priority → 按子模块逐步深入 |

---

### 阶段 1：快速建立全局认知（所有规模必做）

**操作**：TUI 选 `overview` 模式，路径填写工程根目录，点提取。

**耗时**：< 1s（任意规模）

**产出**：`raw/<工程名>/__overview.md`，包含：
- 架构层识别（API层 / 业务层 / 数据层 / UI层）
- 推荐阅读路径（5步，基于配置文件 + 入口文件 + git 热力图）
- 目录结构、配置文件、README、git log、高频变更文件

**Claude 指令**：

```
读取 raw/<工程名>/__overview.md。
用三个层次回答：
1. 一句话：这个工程是什么（类型 + 主要职责）
2. 五分钟：主要模块名称、数据从哪里进、从哪里出、关键文件路径
3. 深度：各模块职责边界、模块间依赖关系、新人最应先读的 3 个文件

规则：只陈述文件中已有的事实，不推断意图，不建议重构。
```

**判断信号**：如果 LLM 能准确说出"这是一个 X 类型的工程，核心入口是 Y，主要处理 Z"，
则阶段 1 完成，进入阶段 2。如果回答模糊，说明 overview 信息不够，直接进入阶段 2。

---

### 阶段 2：建立系统契约认知（中型 / 大型工程）

**操作**：TUI 选 `priority` 模式，提取测试文件 + 类型定义 + 入口文件 + `__imports.json`。

**耗时**：通常 < 5s

**产出**：
- 测试文件（系统契约：应该做什么 / 不该做什么）
- 类型定义文件（模块间数据结构契约）
- 入口文件（系统启动和路由逻辑）
- `__imports.json`（模块依赖关系图）

**Claude 指令 — 找到核心枢纽模块**：

```
读取 raw/<工程名>/__imports.json。
统计每个模块被 import 的次数，列出 Top 10。
这些是系统的依赖枢纽，其他模块都依赖它们。
对每个枢纽模块，说明它在 __overview.md 中属于哪个架构层。
```

**Claude 指令 — 理解系统契约**：

```
读取所有测试文件（*.spec.ts / *.test.ts / test_*.py）。
不读实现代码，只从测试的角度回答：
- 这个系统对外承诺的核心行为有哪些？（列出 Top 10）
- 哪些边界条件被明确处理？
- 哪些重要场景明显没有测试覆盖？（潜在风险区域）
```

**Claude 指令 — 理解数据结构边界**：

```
读取所有类型定义文件（*.d.ts / types.ts / interfaces.py / models.py）。
列出：
- 主要数据实体有哪些？它们之间有什么关系？
- 模块对外暴露的接口是什么？（公开 API 边界）
- 有哪些类型只在内部使用，外部不应该直接依赖？
```

---

### 阶段 3：困惑驱动的定向深入（大型工程 / 准备修改代码前）

阶段 1+2 完成后，LLM 已建立基本认知框架但仍有盲区。
用「困惑驱动」方法让 LLM 自己暴露盲区，再定向补充。

**Claude 指令 — 暴露盲区**：

```
基于已读取的 __overview.md、__imports.json 和测试文件，
作为一个需要修改这个工程的工程师，
列出你还不清楚的 10 个关键问题。
```

|------|------|------|
| CLAUDE.md 要不要包含分析策略？ | 不包含，极简 | 每次 Claude Code 启动都自动加载 CLAUDE.md，分析策略混进去会干扰日常开发辅助任务 |
| 为什么不用多个 CLAUDE.md（每来源一个）？ | 统一一个 | 你的所有内容在同一个项目目录，多个 CLAUDE.md 会冲突；用 Skill 的参数（`--source`）代替多文件 |
| Skill 要不要内嵌提取逻辑？ | 不内嵌 | Skill 只编排，提取是 content-extract CLI 的事。职责分离，Skill 专注分析 |
| wiki 要不要按来源分多个子目录？ | 统一一个 wiki/ | 所有内容在同一 wiki 里，相同概念自动累积置信度，跨来源矛盾点自动暴露。若某个来源需要隔离（如公司内部代码），再单独处理 |
