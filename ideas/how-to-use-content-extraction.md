# How to Use Content Extraction — 操作指南

> 创建日期：2026-06-06 | 持续更新中
>
> **定位**：本文是 [content-extraction.md](./content-extraction.md) 的**使用指南**，专注于「拿到 raw/ 后怎么做」。
> content-extraction.md 是工具手册（怎么获取内容），本文是使用手册（怎么分析内容、构建知识）。
>
> **关联文档**：
> - [content-extraction.md](./content-extraction.md) — 各来源提取工具和脚本
> - [content-extraction-tool-plan.md](./content-extraction-tool-plan.md) — 工程化落地设计

---

## 整体流程图

```
raw/*.md                →           wiki/*.md              →       你的查询
（TUI/CLI 生成的原料）       （Claude Code 整理的结构化笔记）     （自然语言提问）

       ↑                          ↑              ↑
  content-extract            CLAUDE.md        /content-extract
  （本文不涉及）          （极简项目结构声明）   （Skill：所有分析逻辑）
```

**核心分工**：
- `raw/` 是机器生产的，格式一致，人不读
- `wiki/` 是 Claude 整理的，人通过 Obsidian 消费
- `CLAUDE.md` 只声明项目结构（极简，约 15 行），不含分析策略
- `Skill` 包含所有判断逻辑：检测内容类型、选对应策略、困惑驱动等
- 你只做两件事：**策源**（决定抓什么）和**提问**（决定问什么）

# Part 1：基础流程

## 1.1 建立 CLAUDE.md（一次性）

CLAUDE.md 是你和 Claude 之间的「合同」——告诉它你是谁、内容是什么、wiki 该怎么组织。

放在项目目录的根（`raw/` 的上级），Claude Code 启动时自动读取。

### 通用模板

```markdown
# 知识库构建规则

你是这批内容的知识管理员。
原始内容在 `./raw/` 目录（Markdown 格式）。
任务：构建结构化 Wiki 到 `./wiki/`。

## 内容来源识别

raw/ 文件名前缀说明：
- `web__`：网页/博客爬取
- `bili__`：Bilibili 视频转录
- `article__`：单篇网络文章

每个文件的 frontmatter 包含 source / type / platform 等字段。

## Wiki 目录结构

wiki/
├── INDEX.md          # 所有概念索引 + 来源分布图
├── concepts/         # 核心概念（跨来源整合）
│   └── CONCEPT.md
├── by-source/        # 按来源分类的摘要
└── changelog.md

## 每个概念页面格式（wiki/concepts/CONCEPT.md）

---
sources: [文件名列表]
related: [[concept-a]], [[concept-b]]
last-updated: YYYY-MM-DD
---

## 核心定义
（1-3 句话）

## 关键细节
（技术细节、具体数据、重要案例）

## 不同来源的视角
（同一概念在不同视频/文章中的表述差异）

## 未解答的问题
（整理这个概念时仍不清楚的地方）

## 构建步骤

1. 读取所有 `./raw/*.md` 文件
2. 识别跨来源的核心概念，每个建一个 `concepts/CONCEPT.md`
3. 为每种来源类型生成摘要（`by-source/*.md`）
4. 构建 `INDEX.md`：概念列表 + 来源分布 + 关联关系

## 更新规则

新文件加入 raw/ 时：
- 更新受影响的概念页面
- 在 changelog.md 记录
- 如有新概念，添加到 INDEX.md
```

> **建议**：不同内容类型要定制 CLAUDE.md。投资视频和技术文档的整理逻辑完全不同。
> 见 Part 3 各内容类型的具体 CLAUDE.md 配置。

## 1.2 第一次构建 Wiki

```bash
# 进入项目目录
cd /path/to/your/project

# 启动 Claude Code
claude
```

在 Claude Code 里，**先做快速扫描，确认方向再全量建**：

```
第一步（扫描）：
读取 ./raw/ 下所有文件的文件名和 frontmatter，
告诉我：有多少个文件、主要内容类型是什么、你识别出的核心主题有哪些。
不用读正文，只看元数据。
```

确认方向对了，再：

```
第二步（全量建）：
读取 ./raw/ 下所有文件，按照 CLAUDE.md 的规则构建 Wiki 到 ./wiki/。
先建 INDEX.md 框架，再逐个建 concepts/ 页面。
```

## 1.3 日常查询

Wiki 建好后，三种查询方式按需选择：

### 方式 A：直接问 Claude Code（最常用）

```bash
claude
```

```
根据 wiki/，[主题] 的核心逻辑是什么？
```

```
wiki/concepts/ 里哪些概念是相互矛盾的？找出来并解释。
```

```
我想了解 [概念X]，从 wiki 里找出所有相关内容，
给我一个从基础到深入的阅读路径。
```

### 方式 B：Obsidian 浏览（通勤/碎片时间）

把 `wiki/` 目录用 Obsidian 打开：
- Graph view 看概念关联网络
- DASHBOARD.md 看最近更新和知识 gap
- 双击 `[[链接]]` 在概念间跳转

### 方式 C：带上下文的精准查询

```bash
# 先找相关文件
grep -rl "关键词" ./wiki/

# 在 Claude Code 里：
# "读取这几个文件，深入解释 XXX"
```

## 1.4 新内容加入时更新

每次 TUI/CLI 抓完新内容后，在 Claude Code 里：

```
raw/ 有新增文件：[文件名列表]
请更新 wiki/：
- 补充/更新相关 concepts/ 页面
- 在 changelog.md 记录本次新增
- 如有新概念，添加到 INDEX.md
不需要重建整个 wiki，只更新受影响的部分。
```

---

## 1.5 常见问题处理

### 问题：raw/ 文件太多，一次塞不进上下文

Claude 的上下文约 200k token ≈ 600KB 文本。文件过多时分批处理：

```
先只读 raw/ 下前 20 个文件，建立初始 wiki 框架和主要概念列表。
```

```
继续读接下来 20 个文件，补充和更新 wiki。
（已有的 concepts/ 文件你可以直接修改，不需要重建）
```

### 问题：不知道 wiki 里有什么

```
读取 wiki/INDEX.md，告诉我当前 wiki 里有哪些主要概念，
按「出现频次」和「关联数」排序，并标出哪些概念是孤立的（没有 related）。
```

### 问题：wiki 质量参差不齐（概念太碎）

```
读取所有 wiki/concepts/ 文件，
找出哪些概念实际上是同一个东西的不同表述，建议合并列表。
哪些概念只出现在 1 个来源里，标记为「低置信度」。
```

# Part 2：进阶技巧

## 2.1 困惑驱动的第二轮提取

> 适用于：第一轮构建完 wiki 后，发现还有大量「不知道自己不知道」的盲区。

**原理**：wiki 建好后，让 Claude 主动暴露它的盲区，再用这些问题驱动定向提取。比「尽量多抓内容」更精准。

```
第一步：
读取当前所有 wiki/concepts/ 文件。
作为一个需要深度理解这批内容的人，
列出你还不清楚的 10 个关键问题。
格式：问题 → 可能在哪里找到答案（什么类型的内容/来源）
```

Claude 会输出类似：

```
1. 作者对「抄底」和「趋势跟随」的优先级判断是什么？→ 可能在具体策略类视频
2. 这些建议适用于什么仓位规模？有没有提到资金门槛？→ 可能在实战案例类视频
3. 风险管理的具体操作（止损位置、仓位比例）在哪里有详细说明？→ ...
```

```
第二步：
根据问题1，在 raw/ 里搜索相关内容：
grep -rl "抄底\|趋势" ./raw/
把找到的文件读一遍，补充 wiki/concepts/ 里的相关页面。
```

重复直到 Claude 不再产生新问题。

---

## 2.2 跨来源对比分析

当你有多个来源（多个 UP 主、视频 + 文章等）时：

```
读取 wiki/by-source/ 下所有文件，
比较不同来源对「[同一主题]」的观点差异：
- 有哪些一致的观点（可信度高）
- 有哪些明显矛盾的地方（需要我自己判断）
- 哪个来源对这个主题的解释最系统完整
```

---

## 2.3 发现知识 gap

```
读取 wiki/INDEX.md 和 wiki/DASHBOARD.md（如果存在）。
告诉我：
1. 哪些概念有「未解答的问题」（frontmatter 里的该字段不为空）
2. 哪些概念是孤立的（没有 related 连接）
3. 按重要性排序，列出最值得补充的 5 个 gap
```

# Part 3：不同内容类型的处理差异

> 这一 Part 会持续扩充。每种内容类型的特点不同，CLAUDE.md 的配置和查询策略也不同。

---

## 3.1 投资/财经视频（Bilibili）

**内容特点**：
- 口语化转录，有重复和填充词
- 同一概念被多个视频反复提及（信号强度不同）
- 有时效性（市场观点随时间变化）
- 有些视频是「故事化」的，不是「方法论」的

**CLAUDE.md 定制要点**：

```markdown
## 内容特征
- 全部是投资/炒股主题的短视频口语转录
- 字幕有重复行和填充词（已预处理简体中文，但口语特征保留）
- 核心价值：反复出现的观点 > 只提一次的观点
- 注意时效性：涉及具体市场判断的内容标注日期

## 概念分类（wiki/concepts/ 下按这三类组织子目录）
- 理念/哲学类：对投资本质的看法（不受时效影响）
- 方法论类：具体操作策略（需注明适用条件）
- 案例类：具体市场分析（强时效性，标注日期）

## 重要提示
- 「谷票」「谷市」是「股票」「股市」的口语谐音写法，统一处理为「股票/股市」
- 重复出现 3 次以上的观点才建立独立概念页面
- 相互矛盾的观点要在 INDEX.md 底部的「冲突记录」里列出
```

**推荐的第一个查询**：

```
读取所有 bili__*.md 文件，
统计每个核心概念（投资理念/操作策略）在多少个视频中出现过，
按出现频次排序列出 Top 15，这是最值得建 wiki 页面的概念。
```

**特别有用的查询**：

```
这位 UP 主对「散户」的核心判断是什么？
他认为散户最常犯的 3 个错误是什么，在哪些视频里有具体说明？
```

```
把他关于「什么时候该空仓」的所有表述汇总，
看看他的标准是否一致，还是在不同视频里有不同说法。
```

## 3.2 单篇文章（微信公众号 / 今日头条 / 通用网页）

**内容特点**：
- 覆盖微信公众号、今日头条/西瓜、知乎、少数派、掘金、Medium 等各类博客
- 反爬难度差异大：微信最强、头条中等、其他大多数可直接抓
- 适合「刷到一篇好文章，立刻收进知识库」的随手收藏场景

**抓取路由（自动识别，无需手动选）**：

| 平台 | 工具 | 成功率 |
|------|------|--------|
| 微信公众号 | Camoufox stealth browser（模拟 iPhone 微信 UA） | 95%+ |
| 今日头条 / 西瓜 | Playwright Chromium | 90%+ |
| 知乎 / 少数派 / 掘金 / 其他 | crawl4ai → Jina Reader 降级 | 99%+ |

**依赖安装（一次性，如果还没装）**：

```bash
pip install camoufox && camoufox fetch          # 微信专用
pip install playwright && playwright install chromium  # 头条专用
```

---

### TUI 使用

粘贴 URL，来源类型选「自动识别」或「article 单篇文章」，点「提取」即可。微信/头条/通用平台自动路由，无需手动选择。

---

### CLAUDE.md 定制要点

```markdown
## 内容特征
- 单篇文章集合，来源平台混合（微信/头条/知乎等）
- platform frontmatter 标注来源：wechat | toutiao | generic
- 核心价值：观点/方法论提炼，不是段落引用

## wiki 组织原则
- 按「核心观点」组织，不按「文章」组织
- 同一观点在多篇文章出现 → 合并为一个 concepts/ 页面，注明来源数量
- 明显来自同一流派的文章 → 建一个「流派/作者视角」页面
```

**推荐的第一个查询**：

```
读取所有 article__*.md 文件的标题和 frontmatter，
按 platform 分类统计数量，
列出你识别到的 Top 10 高频主题关键词。
```

**特别有用的查询**：

```
这批文章里有没有观点相互矛盾的？
找出同一主题下立场不同的文章，各自的核心论点是什么？
```

```
哪篇文章的信息密度最高？（包含最多可操作的具体建议，而不只是泛泛观点）
```

## 3.3 代码工程

**内容特点**：
- 目标是「理解架构和修改意图」，不是「记录所有代码」
- 大型工程（万行以上）全量提取耗时长且 LLM 上下文放不下，需要分层策略
- 最有价值的信息不是代码行本身，而是**模块间关系**和**历史变更模式**
- 不同规模工程适合不同的提取深度（见 3.7 节通用落地指南）

**三档提取模式**（content-extract TUI 选 `code 本地代码工程`）：

| 模式 | 输出 | 耗时（以 spartacus 为例） | 适用场景 |
|------|------|--------------------------|---------|
| `overview`（默认） | 1 个 `__overview.md` | ~0.3s | 快速建立全局认知 |
| `priority` | ~92 个文件 + `__imports.json` | ~1s | 准备修改代码前 |
| `full` | ~12098 个文件 + `__imports.json` | ~68s | 深度阅读特定模块 |

**`__overview.md` 包含的内容**（overview 模式自动生成）：
- **架构层识别**：启发式匹配目录名，识别 API层 / 业务逻辑层 / 数据访问层 / UI层 / 工具层
- **推荐阅读路径**：基于配置文件、入口文件、git 热力图自动生成 5 步阅读顺序
- 目录结构（3 层深）、配置文件全文、README、git log（最近 30 条）、高频变更文件 Top 20

**`__imports.json` 包含的内容**（priority / full 模式生成）：
- 已提取文件的 import 依赖关系图，格式：`{ "src/auth/login.ts": ["../auth.service", "../../shared/models"] }`
- 让 LLM 能直接看出模块调用链，而不是逐文件猜测

**CLAUDE.md 定制要点**：

```markdown
## 内容特征
- 这是一个 [语言/框架] 代码工程的提取结果
- `__overview.md`：项目全局概览（架构层、阅读路径、git 信息）
- `__imports.json`：模块 import 依赖图（priority/full 模式生成）
- 测试文件（`*.spec.ts` / `test_*.py`）优先阅读——它们是系统契约文档
- 目标：理解架构和核心模块职责，不是记录每行代码

## 分析目标
- 理解系统整体架构（模块边界、数据流向、关键依赖）
- 识别高频变更模块（复杂度热点）
- 找到新开发者需要最先了解的 3-5 个核心文件
```

**推荐的第一个查询**（overview 模式后）：

```
读取 ./raw/<工程名>/__overview.md，
基于其中的架构层识别和推荐阅读路径，
用三个层次回答：
1. 一句话：这个工程是什么
2. 五分钟解释：主要模块、数据流向、核心文件
3. 深度拆解：各模块职责、模块间关系、新人应该先看什么
只陈述文件中有的事实，不推断和建议重构。
```

**priority 模式后的查询**（读测试 + 类型 + imports）：

```
读取 ./raw/<工程名>/__imports.json，
找出被 import 次数最多的 5 个模块，
这些是系统的核心依赖枢纽。
再读这 5 个模块对应的 .md 文件，解释它们各自负责什么。
```

```
读取所有 *.spec.ts / *.test.ts 文件，
列出这个系统对外承诺的核心契约（what it should do）：
- 有哪些主要功能被测试覆盖？
- 有哪些边界条件被明确处理？
- 有哪些明显没有测试覆盖的区域？
```

> **大型工程（>5000 个文件）的处理策略**：见 3.7 节通用落地指南。

## 3.4 电子书（EPUB / PDF / MOBI / AZW3）

**和其他来源最根本的不同**：书是唯一一种「结构本身就是论点」的内容。

- 视频/文章：每条独立，顺序无关
- **书**：第3章建立在第2章基础上，章节顺序是作者精心设计的论证链

这意味着 wiki 组织逻辑不同：**不按章节组织，按核心论点组织**。

---

### 格式处理路径

| 格式 | 处理方式 |
|------|---------|
| EPUB | 直接提取，结构最整齐，推荐 |
| PDF | 直接提取（文字型）或 OCR（扫描版） |
| MOBI / AZW3 | 先用 Calibre 转 EPUB：`ebook-convert input.mobi output.epub` |

---

### 提取后的文件类型

每本 EPUB 书会输出两类文件（`content-extract ebook` 生成）：

| 文件 | 命名 | 何时存在 | 用途 |
|------|------|---------|------|
| 目录结构 | `epub__书名__toc.md` | EPUB 有 TOC 时 | **先读**，理解全书论点框架 |
| 章节全文 | `epub__书名__ch001.md` ... | 始终存在 | **按需读**，用于补充细节 |

> frontmatter 里包含 `chapter_title` 字段，Claude 不读正文就能判断哪些章节最相关。

---

### 四种书型的处理策略

不同类型的书，wiki 组织方式完全不同：

| 书型 | 例子 | wiki 组织方式 | 特别注意 |
|------|------|------------|---------|
| **技术手册/参考书** | 官方文档书籍、API 手册 | 按功能/概念分类，不按章节 | 章节是查找辅助，不是论点单元 |
| **商业/自我提升书** | 投资、管理、效率类 | 核心论点 → 子论点 → 具体操作步骤 | 警惕「激励性语言」≠「可操作建议」 |
| **学术著作** | 研究专著、论文集 | 按研究问题：背景/方法/结论/局限性 | 结论章节信息密度最高，优先读 |
| **叙事性/案例集** | 商业传记、案例研究 | 按规律/模式提炼，不按故事 | 多个案例讲同一规律时合并为一个概念页 |

---

### 两阶段读取策略（避免上下文超限）

书通常有几十章，一次全部塞入会超出上下文。标准两阶段：

**第一阶段：建立全书认知框架（约 5-10 分钟）**

```
读取以下文件（如果存在）：
1. epub__书名__toc.md（目录结构）
2. epub__书名__ch001.md（前言/序）
3. 最后 1-2 个章节文件（结语/总结）

基于这些内容：
- 用一句话说明这本书的核心主张
- 列出全书的 5-8 个核心论点（不要超过 10 个）
- 标注哪些章节对应哪些论点
- 这本书属于哪种类型（技术手册/商业/学术/叙事）
```

**第二阶段：定向深读（按需）**

```
我想深入了解「[论点X]」，根据 TOC 映射，它对应章节 ch003-ch005。
读取这三个章节文件，提炼：
- 核心观点（1-3 句话）
- 关键证据/案例（原文引用）
- 与其他论点的关联
- 仍不清楚的地方
```

### CLAUDE.md 定制要点

```markdown
## 内容特征
- EPUB/PDF 书籍，已按章节切分为独立 md 文件
- `epub__书名__toc.md`：目录结构，先读此文件建立框架
- `epub__书名__ch001.md` 等：各章节全文，frontmatter 含 chapter_title 字段
- 书型：[技术手册 / 商业/自我提升 / 学术著作 / 叙事/案例集]（填入实际类型）

## wiki 组织原则
- 按「核心论点」组织，不按「章节」组织
- 技术手册：按功能/概念分类
- 商业书：区分「原则（说了不代表可操作）」和「操作（有具体步骤）」
- 学术著作：保留「局限性」字段，记录结论的适用边界
- 重要原文引用用 > 引用格式保留，标注章节号

## 概念页面额外字段
- `book_type:`（技术/商业/学术/叙事）
- `chapter_refs:`（对应原书哪些章节）
- `is_actionable: true/false`（商业书专用：是否有可操作步骤）
```

### 借鉴自 book-to-skill 的三项增强（2026-06-15 更新）

参考：[virgiliojr94/book-to-skill](https://github.com/virgiliojr94/book-to-skill)，该工具将书籍提炼为 LLM 可调用的 skill，其设计哲学对本流程有三点可借鉴之处。

---

#### 增强一：每个 is_actionable 概念页补充「典型应用示例」字段

**位置**：在「不同来源的视角」之前插入。

**内容要求**：重建作者书中实际走过的一个完整例子——不是「白纸学习法有4个步骤」，而是「作者备考心力衰竭章节时，具体怎么用白纸法，写了什么，遗漏了哪里，第二天怎么补」。

**why**：概念定义告诉你「什么是白纸法」，典型应用示例告诉你「在真实场景里它长什么样」。后者才是回看时真正有价值的东西。对于 `is_actionable: false` 的原则性概念，此字段可省略。

**格式参考**：

```markdown
## 典型应用示例

**[场景名称]**

[作者实际操作的完整过程，包含：输入（面对什么内容）、行动（具体怎么做）、输出（得到什么结果）、关键感受/反直觉发现]
```

**已应用的示例**：`wiki/learning/` 下所有 6 个 is_actionable 概念页均已补充此字段，见 [[learning/concepts/learning-主动回忆]] 等。

---

#### 增强二：每个领域补充 cheatsheet.md

**位置**：与 INDEX.md、by-source/ 并列，放在领域根目录。

**内容要求**（按优先级排序，参考 book-to-skill 的定义）：

1. **决策规则**：「当 X 情境时，用 Y，因为 Z。」每行都应帮助读者做决定，而不只是描述概念。
2. **权衡矩阵**：竞争方案在作者关心的维度上的评分，让读者在自身约束下选择。
3. **阈值与默认值**：作者明确给出的具体数字或规则（「睡眠 ≥ 7 小时」「复习间隔 = 考试前1/5时间」）。
4. **Tells & Smells**：快速识别「我现在陷入什么陷阱」的启发式（「你有这个感觉 → 实际情况 → 应该做什么」）。

**什么不应该放**：纯术语定义（那是 concepts/ 的事）、长段解释性文字（那是来源摘要的事）。

**已应用的示例**：见 [[learning/cheatsheet]] — 覆盖方法选择决策表、阶段路由、关键阈值、Tells & Smells、适用边界。

---

#### 增强三：对话式查询 wiki 时的 Discovery Loop Tax 意识

**背景**：book-to-skill 测量了让 LLM agent 实时读 PDF 回答问题的成本：一本 256K token 的书，每次问答消耗约 77K tokens（discovery loop 方式）；而预结构化的 skill 只需约 5K tokens（24-51倍差距）。

**本流程的现状**：wiki/ 是静态文件，通过 Obsidian 阅读，不存在 discovery loop 问题。

**什么时候需要注意**：如果未来实现「用 Claude Code 对话查询 wiki」的 agent（如 `/content-extract query 什么是仓位管理`），不要直接把整个 INDEX.md + 所有 concepts/ 文件塞进上下文。正确做法：

```
查询时的上下文策略：
1. 先加载领域 INDEX.md（~2K token），定位相关概念名
2. 按需加载对应的 concepts/CONCEPT.md（~1K token）
3. 如需决策辅助，加载 cheatsheet.md（~1.5K token）
总计约 4-5K token，而非全量加载 wiki/ 所有文件（>50K token）
```

这与 wiki/ 的分层目录结构（INDEX → concepts/ → cheatsheet）完全对应，设计天然支持按需加载，无需改造。

当你有同一主题的多本书时：

```
读取以下书的 toc.md：
- epub__书名A__toc.md
- epub__书名B__toc.md

这两本书在「[主题]」上的核心主张是什么？
有哪些一致的观点？有哪些明显的分歧？
```

## 3.8 Topic 学习模式（主题驱动，跨来源混合）

**和其他所有模式的根本差异**：

其他模式是**来源驱动**的——你知道你要从哪里抓，目录名就是来源（UP 主名、网站域名、书名）。

Topic 模式是**学习目标驱动**的——你想学「量化投资」或「Rust 异步编程」，然后去找资料，资料可能来自任何地方：一个 UP 主的视频系列 + 一篇技术博客 + 一本电子书 + 一个 GitHub 仓库。

**Topic 是容器，不是来源类型。** 容器里的内容仍然是 web/video/ebook/code 等现有类型，只是多了一个「为什么收集这些」的上下文。

---

```
│   │   ├── .processed.json

│   │   ├── web__xxx.md            ← 网页来源

│   │   ├── epub__书名__ch001.md   ← 电子书章节
│   │   └── local__笔记.md         ← 本地文件（不需要网络）
```

### Topic frontmatter 额外字段

每个归入 topic 的文件，frontmatter 多两个字段：

```yaml

source: https://...
type: video
platform: bilibili
topic: "量化投资入门"            # 所属 topic
topic_role: "策略原理介绍"       # 这个资料在 topic 里的定位（可选）
---
```

### 主题角色（topic_role）详解

`topic_role` 是给每个资料贴的**定位标签**，不影响抓取行为，影响的是后续 wiki 分析质量——Claude 会根据角色决定「这份资料该怎么用」。

| 角色 | 适合放什么 | 举例 |
|------|-----------|------|
| **入门概述** | 第一篇应该读的，给你建立整体框架 | 维基百科词条、官方 Getting Started 文档、某个「什么是 X」视频 |
| **核心方法论** | 系统讲原理和方法的，有深度 | 经典教程、权威书籍、技术规范文档 |
| **深度参考** | 不用顺序读，遇到问题才查的 | API 文档、手册、进阶专题文章 |
| **代码实例** | 有可运行代码的实践资料 | GitHub 仓库、代码教程、示例项目 |
| **案例研究** | 讲真实案例/经验的 | 踩坑博客、实战复盘、行业案例 |
| **工具介绍** | 介绍某个具体工具怎么用的 | 某库的使用教程、工具对比文章 |
| **个人笔记** | 你自己写的笔记、整理的内容 | 本地 md 文件、读书笔记 |
| **不指定** | 懒得分类，或者不确定放哪 | 随手收藏，之后再说 |

**角色如何影响 wiki 分析：**

1. **推荐学习路径**：Claude 会按角色建议阅读顺序——先看「入门概述」建立框架，再读「核心方法论」深入理解，遇到问题查「深度参考」，动手时参考「代码实例」。

2. **内容置信度加权**：「核心方法论」里的概念置信度更高；「案例研究」的观点会标注"来自实践经验"；「个人笔记」会被识别为"已有认知"而不是新输入。

3. **盲区识别**：Claude 扫描 topic_role 分布后，能直接指出「你目前没有核心方法论类资料，建议补充一本系统性的书」这类缺口。

**实际使用建议：**

- 刚开始收集时，明显是「入门概述」的打上标签，其他先选「不指定」
- 本地笔记固定选「个人笔记」，让 Claude 区分你的已有认知和新输入
- GitHub 仓库选「代码实例」，分析时会当代码参考而非理论资料
- 不确定时选「不指定」比选错更好，Claude 会根据内容自行判断

### 线上资料 vs 本地资料的处理差异

| 资料类型 | TUI 操作 | 处理方式 |
|---------|---------|---------|
| 在线网页 | 粘贴 URL，选类型，填 topic | 正常爬取，归入 `raw/topics/<topic>/` |
| 在线视频 | 粘贴 URL，选 video，填 topic | 正常下载/转录，归入 topic 目录 |
| 本地 epub/pdf | 浏览选文件，填 topic | 直接格式转换，不需要网络 |
| 本地 md/html | 浏览选文件，填 topic | 直接读取，写入 frontmatter，**不复制原文件** |

**本地 md/html 的特殊处理**：不应该复制文件，应该只生成一个「引用文件」，frontmatter 里记录原始路径，正文是原文件内容的提取结果。这样避免数据重复，也保持原文件的单一来源。

```yaml
---
source: /Users/I340818/Documents/笔记/量化策略笔记.md
type: local_doc
topic: "量化投资入门"
topic_role: "个人笔记"
---
```

### Wiki 分析：学习路径导向

Topic 的 wiki 构建目标和其他模式不同——不是「这些资料说了什么」，而是「我怎么学会这个主题」。

**CLAUDE.md 定制要点**：

```markdown
## 内容特征
- 这是「[topic]」主题的学习资料集合
- 来源混合：视频转录 + 网页文章 + 电子书章节 + 本地笔记
- 每个文件的 topic_role 字段说明它在这个主题里的定位

## 分析目标（不同于来源驱动模式）
- 不是整理「每个来源说了什么」
- 而是构建「这个主题的知识地图」：
  1. 核心概念和它们的关系
  2. 推荐学习路径（从哪里开始，到哪里结束）
  3. 各来源的定位（这个视频适合理解 X，这本书适合深入 Y）
  4. 当前资料的盲区（这个主题还缺什么）

## wiki 组织
- wiki/topics/[topic名]/concepts/  ← 核心概念
- wiki/topics/[topic名]/roadmap.md ← 推荐学习路径
- wiki/topics/[topic名]/sources.md ← 各来源的定位说明
- wiki/topics/[topic名]/gaps.md    ← 当前资料的知识盲区
```

**推荐的第一个查询**：

```
读取 raw/topics/[topic]/ 下所有文件的文件名和 frontmatter（不读正文）。
告诉我：
1. 这个 topic 目前有哪些来源类型？各多少个？
2. 按 topic_role 分类，分别是什么定位？
3. 从覆盖面看，这个学习集合的主要盲区在哪里？
```

**全量构建**：

```
读取 raw/topics/[topic]/ 下所有文件，
按照 CLAUDE.md 的规则，构建「学习路径导向」的 wiki：
- 先建 roadmap.md（推荐学习顺序）
- 再建 concepts/（核心概念，跨来源整合）
- 再建 sources.md（各来源的定位）
- 最后建 gaps.md（当前资料的知识盲区）
```

---

### 与来源驱动模式的对比

| 维度 | 来源驱动（原有） | Topic 学习模式 |
|------|--------------|--------------|
| 组织逻辑 | 按来源（UP主/网站/书名） | 按学习目标 |
| 适合场景 | 消化某个来源的全部内容 | 学习某个主题 |
| Wiki 目标 | 这个来源说了什么 | 我怎么学会这个主题 |
| 来源数量 | 通常单一来源 | 混合多个来源 |
| 可信度 | 均等（同一来源） | 需要按类型加权 |
| 本地文件 | 不常见 | 核心使用场景 |

两种模式**并列存在，互不干扰**。同一个视频可以同时存在于来源驱动目录（完整消化该 UP 主）和 topic 目录（学习某主题时的参考）——用 `topic` frontmatter 字段标记即可，不需要复制文件。

## 3.7 通用代码工程落地指南

> 以下流程适用于任何代码工程（Angular/TypeScript、Python、Go、Java 等），
> 将「抓取 → 理解 → 深入」分为四个阶段，按工程规模选择进入深度。

---

### 阶段 0：判断工程规模，选择策略

先在 TUI 里用 `overview` 模式跑一次（~0.3s），读 `__overview.md` 中的目录结构和文件数量，
然后按下表选择后续策略：

| 工程规模 | 文件数量参考 | 推荐策略 |
|---------|------------|---------|
| **小型**（脚本/库） | < 500 个文件 | overview → 直接 full → wiki 构建 |
| **中型**（服务/应用） | 500–5000 个文件 | overview → priority → 困惑驱动定向补充 |
| **大型**（平台/框架） | > 5000 个文件 | overview → priority → 按子模块逐步深入 |

---

### 阶段 1：快速建立全局认知（所有规模必做）

**操作**：TUI 选 `overview` 模式，路径填写工程根目录，点提取。

**耗时**：< 1s（任意规模）

**产出**：`raw/<工程名>/__overview.md`，包含：
- 架构层识别（API层 / 业务层 / 数据层 / UI层）
- 推荐阅读路径（5步，基于配置文件 + 入口文件 + git 热力图）
- 目录结构、配置文件、README、git log、高频变更文件

**Claude 指令**：

```
读取 raw/<工程名>/__overview.md。
用三个层次回答：
1. 一句话：这个工程是什么（类型 + 主要职责）
2. 五分钟：主要模块名称、数据从哪里进、从哪里出、关键文件路径
3. 深度：各模块职责边界、模块间依赖关系、新人最应先读的 3 个文件

规则：只陈述文件中已有的事实，不推断意图，不建议重构。
```

**判断信号**：如果 LLM 能准确说出"这是一个 X 类型的工程，核心入口是 Y，主要处理 Z"，
则阶段 1 完成，进入阶段 2。如果回答模糊，说明 overview 信息不够，直接进入阶段 2。

---

### 阶段 2：建立系统契约认知（中型 / 大型工程）

**操作**：TUI 选 `priority` 模式，提取测试文件 + 类型定义 + 入口文件 + `__imports.json`。

**耗时**：通常 < 5s

**产出**：
- 测试文件（系统契约：应该做什么 / 不该做什么）
- 类型定义文件（模块间数据结构契约）
- 入口文件（系统启动和路由逻辑）
- `__imports.json`（模块依赖关系图）

**Claude 指令 — 找到核心枢纽模块**：

```
读取 raw/<工程名>/__imports.json。
统计每个模块被 import 的次数，列出 Top 10。
这些是系统的依赖枢纽，其他模块都依赖它们。
对每个枢纽模块，说明它在 __overview.md 中属于哪个架构层。
```

**Claude 指令 — 理解系统契约**：

```
读取所有测试文件（*.spec.ts / *.test.ts / test_*.py）。
不读实现代码，只从测试的角度回答：
```

## 5.4 为什么这样拆分（设计决策记录）

| 问题 | 决策 | 原因 |
|------|------|------|
| CLAUDE.md 要不要包含分析策略？ | 不包含，极简 | 每次 Claude Code 启动都自动加载 CLAUDE.md，分析策略混进去会干扰日常开发辅助任务 |
| 为什么不用多个 CLAUDE.md（每来源一个）？ | 统一一个 | 你的所有内容在同一个项目目录，多个 CLAUDE.md 会冲突；用 Skill 的参数（`--source`）代替多文件 |
| Skill 要不要内嵌提取逻辑？ | 不内嵌 | Skill 只编排，提取是 content-extract CLI 的事。职责分离，Skill 专注分析 |
| wiki 要不要按来源分多个子目录？ | 统一一个 wiki/ | 所有内容在同一 wiki 里，相同概念自动累积置信度，跨来源矛盾点自动暴露。若某个来源需要隔离（如公司内部代码），再单独处理 |
