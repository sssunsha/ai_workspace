## 对外承诺的行为（来自测试文件）

以下契约来源于 `raw/spartacus/` 中的 spec 测试文件，代表系统已验证的行为承诺。

---

### 状态层（NgRx Store）

**数据加载策略**
- 当 Store 中已有实体数据时，Service 层的 `get()` / `getList()` 应该直接返回缓存，不重复发起 HTTP 请求。
- 当 Store 中没有数据时，`get()` / `getList()` 应该派发对应的 `Load*` Action，触发 Effect 异步加载。

**写操作后的缓存清除**
- 当 Create / Update 操作成功或失败后，系统应该额外派发 `OrganizationClearData` Action，确保组织相关的所有缓存被清除，防止过期数据残留。

**列表分页缓存键**
- 当创建分页列表查询 Action 时，系统应该将分页参数序列化为 query string 作为列表缓存的唯一 key，不同分页参数互不覆盖。

**实体更新语义**
- 当 CRUD 成功时，Reducer 应该用响应数据完整替换 Store 中的对应实体（replace）。
- 当关联关系（Assign/Unassign）操作成功时，Reducer 应该将响应数据与现有实体状态**合并**（merge），仅覆盖 `selected` 等变更字段，不丢失其他字段。

**加载状态跟踪**
- 当实体加载成功后，`getLoadingStatus()` 应该返回 `SUCCESS` 状态及 item 数据。
- 当实体加载失败后，`getLoadingStatus()` 应该返回 `ERROR` 状态及 `undefined` item。

**未定义 ID 容错**
- 当 Action payload 中实体 ID（如 `customerId`、`code`）为 `undefined` 时，系统应该用空字符串兜底作为 entity key，不抛出异常。

---

### 路由守卫层（Guards）

**管理员权限守卫（AdminGuard）**
- 当用户角色包含 `ADMIN` 时，应该允许路由激活（返回 `true`）。
- 当用户角色不含 `ADMIN` 时，应该拒绝访问，跳转到 `organization` 路由，并通过全局消息服务展示权限不足的警告。

**用户管理守卫（UserGuard）**
- 当 `isUpdatingUserAllowed()` 返回 `false` 时，应该返回指向 `/organization` 的 `UrlTree` 并展示"notExist"警告消息。

**单元级订单查看守卫（UnitLevelOrdersViewerGuard）**
- 当用户角色包含 `UNITORDERVIEWER` 权限时，应该允许路由激活。
- 当用户角色缺少该权限时，应该跳转到 `home` 路由并展示权限不足警告消息。

---

### HTTP 错误处理层（Interceptors）

**400 Bad Request 处理（OrganizationBadRequestHandler）**
- 当 HTTP 400 响应的错误类型为 `ModelSavingError` 且包含 `B2BUnitModel` 关键字时，应该展示 `conflict.unit` 消息并附带冲突的 `uid`。
- 当错误包含 `B2BCostCenterModel` 关键字时，应该展示 `conflict.costCenter` 消息并附带冲突的 `code`。
- 当错误类型为 `DuplicateUidError`（Permission 重复）时，应该展示 `conflict.permission` 消息并附带冲突的 `code`。
- 当错误无法匹配已知类型时，应该展示 `conflict.unknown` 消息。
- 当响应体缺少 errors 数组时，应该静默处理，不展示任何消息。

**409 Conflict 处理（OrganizationConflictHandler）**
- 当 HTTP 409 响应的错误消息匹配 Budget / User / UserGroup / Unit / CostCenter 关键字时，应该展示对应分类的 `conflict.*` 错误消息，并携带实体的标识字段（code / uid / email）。

**Blob 错误拦截器（BlobErrorInterceptor）**
- 当浏览器环境中 HTTP 响应以 blob 格式返回错误时，应该将 blob 中的 JSON 内容解析并重新包装为 `HttpErrorResponse`，确保上层可以访问到实际错误结构。
- 当在 SSR（非浏览器）环境中时，应该不介入处理（绕过拦截器逻辑）。

### Connector 层（连接器透传契约）

**透传原则**
- 当调用任意 Connector 的方法时（如 `get`、`getList`、`create`、`update`），系统应该将所有入参**原样**透传给底层 Adapter，不做任何转换或过滤。
- Connector 是接口层，不包含任何业务逻辑。

**AccountSummaryConnector 特定契约**
- 当调用 `getDocumentAttachment(userId, orgUnitId, documentId, attachmentId)` 时，应该将全部四个参数原样传递给 Adapter。

**OrgUnitConnector 特定契约**
- 当调用 `assignRole` / `unassignRole` 时，入参为 `(userId, orgCustomerId, roleId)`（注意第二个参数是 `orgCustomerId` 而非 `orgUnitId`）。
- 当调用 `assignApprover` / `unassignApprover` 时，入参为 `(userId, orgUnitId, orgCustomerId, roleId)` 四个参数。

---

### OCC Adapter 层（HTTP 调用契约）

**UnitOrder OCC Adapter**
- 当调用 `loadUnitOrderHistory(userId, pageSize)` 时，应该向 `unitLevelOrderHistory` 端点发送 GET 请求，`pageSize` 以字符串形式传入 query params。
- 当传入 `filters` 为空字符串时，应该从 query params 中排除 filters 参数（不传空值）。
- 当调用 `loadUnitOrderDetail(userId, orderId)` 时，应该向 `unitLevelOrderDetail` 端点发送 GET，响应通过 `ORDER_NORMALIZER` Converter 规范化。

**AccountSummary OCC Adapter**
- 当调用 `getDocumentAttachment` 时，HTTP 请求的 `responseType` 必须为 `blob`，不能是 `json`。

---

### Service/Facade 层（业务逻辑契约）

**用户角色约定（B2BUserService）**
- 当调用 `getAllRoles()` 时，应该始终按固定顺序返回 `[CUSTOMER, MANAGER, APPROVER, ADMIN]` 四种角色，顺序不可变。

**组织单元子节点查询（OrgUnitService）**
- 当传入不存在的 unitId 调用 `getChildUnits()` 时，应该返回空数组，不抛出错误。
- 当调用 `getActiveUnitList()` 时，应该过滤掉 inactive 单元，并按 id 字母升序排列。

**匿名用户守护**
- 当 `UserIdService` 抛出错误（用户为匿名）时，UnitOrderService 的 `loadOrderList` 应该静默拒绝，不派发任何 Store Action。

---

### 组件层（UI 行为契约）

**AccountSummaryDocumentComponent**
- 当用户切换分页或排序时，应该将页码重置为 0 后再重新请求（防止跨页排序后的数据错位）。
- 当用户点击附件下载时，应该调用 `FileDownloadService` 触发浏览器原生下载，而非直接打开 URL。

**AccountSummaryDocumentFilterComponent**
- 当用户选择日期区间筛选维度时，应该在前端控件格式（YYYY-MM-DD）与后端格式（MM/DD/YYYY）之间自动转换。
- 当用户点击清空按钮时，应该将表单重置为「文档编号筛选」默认模式，并立即发出携带默认值的筛选事件（不等待用户点搜索）。

**UserRegistrationFormComponent**
- 当表单无效时用户提交，应该阻止调用 `registerUser`，不发起注册请求。
- 当注册接口返回错误时，应该通过 `GlobalMessageService` 展示错误类型的全局消息，并隐藏加载 spinner。

**AccountSummaryPageMetaResolver**
- 当处于详情页（`orgAccountSummaryDetails`）时，面包屑层级为：Home → Organization → Account Summaries → 详情页（共 4 级）。
- 当处于列表页（`orgAccountSummary`）时，面包屑仅追加「Organization」一级。
