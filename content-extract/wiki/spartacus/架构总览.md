## 一句话描述

Spartacus（官方名：SAP Commerce Cloud Composable Storefront）是基于 Angular 的前端电商店面框架，完全通过 SAP Commerce OCC REST API 与后端通信，以可发布的 npm 库形式分发，支持在不 fork 源码的前提下通过模块组合方式进行扩展。

---

## 主要模块划分

| 模块组 | 顶层路径 | 职责 | 关键文件 |
|--------|----------|------|----------|
| 核心库（core-libs） | `core-libs/core/`<br>`core-libs/storefront/`<br>`core-libs/schematics/`<br>`core-libs/setup/` | 提供全局状态管理、路由、认证、用户会话、CMS 渲染框架、I18N、功能开关（feature toggles）、NgRx store 基础设施；Schematics 负责 `ng add` 安装流程 | `public_api.ts`（核心导出入口）、`base-storefront.module.ts` |
| 功能库（feature-libs） | `feature-libs/` 下各子目录 | 按业务域分包：购物车（cart）、结账（checkout）、订单（order）、用户（user）、产品（product）、组织（organization）、ASM 辅助销售、报价（quote）、追踪（tracking）等，每个业务库进一步分为 root / core / occ / components / assets 五层 | 各库的 `public_api.ts`、`*.module.ts` |
| 集成库（integration-libs） | `integration-libs/` 下各子目录 | 与 SAP 生态系统外部服务的集成点：OPF（支付框架）、CDC（客户数据云）、CDP（客户数据平台）、CDS（商务数据解决方案）、EPD Visualization、S4OM（S/4HANA 订单管理）、OMF、CPQ 报价、Punchout、Segment Refs 等 | 各集成库的 `*.module.ts`、`public_api.ts` |
| 应用程序（projects） | `projects/storefrontapp/`<br>`projects/storefrontapp-e2e-cypress/`<br>`projects/ssr-tests/` | 参考实现的 Angular 应用、Cypress E2E 测试套件、SSR 集成测试 | `spartacus-features.module.ts`（高频变更核心文件） |
| 构建工具（tools） | `tools/` | 内部脚本：breaking-change 检测、ESLint 自定义规则、配置一致性检查（tsconfig 路径 + 依赖版本）、schematics 工具函数 | `tools/config/index.ts`、`tools/eslint-rules/index.ts` |

## 每个功能/集成库的内部分层

所有 feature-libs 和 integration-libs 内部遵循统一的五层结构：

```
<lib>/
  root/        → Angular 模块入口（NgModule + providers），供应用级懒加载注册
  core/        → NgRx 状态管理（actions/reducers/selectors/effects）+ Connector 门面 + Service 业务逻辑 + Guards
  occ/         → OCC HTTP Adapter 实现，负责 URL 构建、序列化、Converter 调用
  components/  → Angular 组件（UI 层），通过 Service 层与状态交互
  assets/      → I18N 翻译文件
```

## 数据流向

**前端到后端（读取场景）：**
用户触发 UI 事件 → Component 调用 Service（facade）→ Service 检查 NgRx Store 缓存 → 缓存未命中时 dispatch Load Action → Effect 拦截 Action → 调用 Connector.get() → Connector 委托 OCC Adapter → Adapter 发起 HTTP GET → 响应通过 NORMALIZER Converter 规范化 → Effect dispatch Success Action → Reducer 更新 Store → Selector 将数据流给 Component

**写操作场景（Create / Update）：**
Component → Service dispatch Create/Update Action → Effect 调用 Connector.create/update() → OCC Adapter 发起 HTTP POST/PATCH → 成功后 Effect dispatch Success Action + OrganizationClearData（清缓存）→ Reducer 更新实体状态

**CMS 驱动的内容渲染：**
路由变化 → CMS 服务从 OCC 拉取页面结构 → CMS 模块动态映射组件类型 → 渲染对应 Angular 组件

## 核心设计决策

1. **库而非 Fork 的扩展模型**
   Spartacus 以 npm 库形式分发，消费者通过 Angular 的依赖注入和模块系统替换/扩展任意 service、component、adapter，而不需要 fork 源码。这保证了后续版本升级路径畅通，代价是扩展时需要理解 DI token 体系。

2. **OCC 独占通信约定**
   所有数据读写均通过 SAP Commerce OCC REST API，没有其他后端通道。Connector/Adapter 分层是对 OCC 依赖的显式隔离：Connector 是业务无关的接口，Adapter 是 OCC 具体实现，可被替换（例如自定义后端）。

3. **NgRx 全量状态 + 乐观缓存策略**
   所有远程数据落入 NgRx Store，Service 层在 dispatch load 前先检查 Store 是否已有数据（惰性加载）。写操作后强制 dispatch `OrganizationClearData` 清除缓存，保证数据一致性而非乐观更新。

4. **Feature Toggles 作为渐进式演进机制**
   `projects/core/src/features-config/feature-toggles/config/feature-toggles.ts` 是变更最频繁的文件（48 次/6 个月），说明功能开关是主要的渐进发布手段，新功能默认关闭，通过 toggle 逐步启用。

5. **Nx Monorepo + 构建标签约束**
   通过 Nx tag 强制执行依赖方向：UI 层不能依赖 Feature 层，Feature 层不能依赖 Integration 层，Core 层不能依赖 UI/Feature 层。违反约束会在 lint 时报错，防止循环依赖。

6. **所有组件和指令使用 `cx` 前缀**
   ESLint 规则强制所有 Angular 组件选择器以 `cx-` 开头，所有指令以 `cx` 前缀 camelCase 命名，保证命名空间隔离和可识别性。

7. **注入依赖使用 `protected` 而非 `private`**
   ESLint 自定义规则禁止在类中将注入的依赖声明为 `private`（FeatureConfigService 除外），强制使用 `protected`，确保消费者可以通过继承扩展服务行为。
