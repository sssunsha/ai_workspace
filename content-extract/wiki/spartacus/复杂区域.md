## 已知复杂模块

### spartacus-features.module.ts — 功能注册中枢

**为何复杂：** 所有 feature-libs 和 integration-libs 的 NgModule 都在此统一注册，包含所有 lazy-loaded 功能模块的 imports 声明。每次新增或移除集成功能，或调整模块加载顺序，都必须修改此文件。文件本身是配置型代码而非逻辑型，但体积大、合并冲突频繁。

**修改风险：** 中等。NgModule 注册错误不会在编译时报错，只会在运行时导致功能静默丢失或重复注册。需要有完整的功能集 E2E 测试覆盖来验证变更。

---

### feature-toggles.ts — 功能开关配置

**为何复杂：** 这是 Spartacus 渐进式功能发布的核心控制点，近 6 个月变更 48 次，是逻辑文件中变更最频繁的。每个开关都可能影响多个下游组件和服务的行为，且开关之间可能存在隐式依赖（A 开启才能开启 B）。

**修改风险：** 高。启用/禁用 toggle 会影响生产用户的功能可用性。Toggle 的生命周期管理（从"默认关"到"默认开"到"删除"）需要跨版本协调，且难以追踪哪些 toggle 组合已在生产环境验证过。未解答：目前是否存在 toggle 之间的依赖图谱文档？

---

### Organization 模块（administration / account-summary / unit-order）

**为何复杂：** B2B 组织管理是 Spartacus 功能密度最高的区域，包含：
- 多层实体关系（OrgUnit → User → Role → Permission → Budget → CostCenter → UserGroup）
- 复杂的 NgRx 状态切片（每个实体类型都有独立的 actions/reducers/selectors/effects）
- 双向关联关系管理（assign/unassign 操作影响多个实体的 Store 状态）
- HTTP 错误类型区分（400 vs 409，以及子类型的关键字匹配逻辑）

**修改风险：** 高。关联关系变更时 Reducer 采用"合并"而非"替换"语义，修改 Reducer 时容易引入字段覆盖 bug。Effects 统一触发 `OrganizationClearData` 的机制依赖约定而非类型约束，遗漏会导致 UI 显示过期数据。

---

### OCC Adapter 层（occ/ 目录）

**为何复杂：** 所有 feature-libs 和 integration-libs 都有自己的 OCC Adapter，负责：
- URL 模板构建（含 path params 和 query params）
- 参数类型转换（如 `pageSize` 需转为字符串）
- 条件性 query param 过滤（空值不传）
- responseType 选择（`json` vs `blob`）
- Converter（NORMALIZER/SERIALIZER）管道调用

每个 Adapter 的实现细节不同，且没有强制的接口约束来保证一致性，容易出现遗漏参数或错误 responseType 的 bug。

**修改风险：** 中等。Adapter 层有单元测试覆盖，但 URL 构建逻辑的测试通常依赖字符串匹配，URL 模板变更后容易漏测 path param 顺序问题。

---

### Blob 错误处理（BlobErrorInterceptor）

**为何复杂：** 浏览器 Fetch API 在返回 blob 类型响应时，HTTP 错误状态码的响应体已被消费为 Blob，无法通过 Angular 的标准 `HttpErrorResponse` 读取 JSON 错误体。拦截器需要将 Blob 解析回 JSON，这涉及异步操作和 SSR/浏览器环境双分支处理。

**修改风险：** 低（代码量小），但影响范围不明显：附件下载相关的所有 HTTP 错误都依赖此拦截器正确工作，出错时 UI 只会收到空错误消息而不报错。

## 已识别的高潜在变更区域（推断）

以下区域未出现在 git 热力图（热力图仅覆盖近 6 个月），但从代码结构判断具有较高修改频率：

- **OPF（Open Payment Framework）集成库**：integration-libs/opf 包含大量子模块（payment、checkout、cta、quick-buy、tokenisation 等），OPF 是近期活跃集成方向，变更预期持续。
- **Schematics 依赖声明**：`dependencies.json` 随每个库升级自动同步，新增集成库时需手动维护，容易遗漏。
- **E2E Cypress 配置和类型扩展**：`continuum.d.ts` 定义了无障碍测试命令扩展，如果 Continuum SDK 升级 API，需要同步更新此类型声明。
